# 需求文档

## 项目描述（输入）
i18n 扩展 + Geo 优化：中文支持、IP 地区检测、货币本地化、支付适配、hreflang、合规

## 需求

### 需求 1：中文语言支持
**目标：** 作为最终用户，我希望能够以简体中文（zh）使用整个应用，以便中文用户获得与英文用户同等的本地化体验。

#### 验收标准
1. 当用户访问 `/zh/` 前缀的路由时，系统应正确渲染所有 UI 文本为简体中文。
2. 当 `zh` 语言被激活时，`src/messages/zh.json` 翻译文件应包含与 `en.json` 相同结构的所有翻译 key。
3. 系统应支持 `en`、`es`、`fr`、`zh` 四种语言，next-intl 路由配置中应包含 `zh` locale。
4. 当新的 UI 文本被添加到任意语言文件时，系统应同步要求在 `zh.json` 中也添加对应翻译。
5. 服务端组件应使用 `getTranslations` 获取中文翻译，客户端组件应使用 `useTranslations`。

---

### 需求 2：翻译 Key 命名规范
**目标：** 作为开发者，我希望有统一的翻译 key 命名规范，以便团队协作时翻译文件保持一致性和可维护性。

#### 验收标准
1. 系统应采用 `模块.组件.元素` 三级命名结构（例如 `auth.login.title`、`dashboard.header.welcome`）。
2. 当翻译 key 以 `common.` 开头时，该 key 应代表跨模块共用的通用文本（如按钮标签、错误消息）。
3. 翻译 key 应使用 camelCase 命名（例如 `auth.loginForm.submitButton`）。
4. 当翻译 key 对应动态插值时，系统应使用 `{variableName}` 语法（例如 `"欢迎，{name}！"`）。
5. 系统应在 `src/messages/` 目录下为每个语言维护独立 JSON 文件，文件结构应与命名规范保持一致。

---

### 需求 3：基于 IP 的地区自动语言检测
**目标：** 作为最终用户，我希望在首次访问时自动跳转到与我所在地区匹配的语言版本，以便无需手动切换语言。

#### 验收标准
1. 当用户首次访问应用根路径时，系统应通过请求的 IP 地址或 `Accept-Language` 标头检测用户所在地区。
2. 当检测到用户来自中国大陆（CN）、台湾（TW）或香港（HK）时，系统应将默认语言设为 `zh`。
3. 当用户已在 Cookie 中保存语言偏好时，系统应优先使用 Cookie 中的语言设置而非 IP 检测结果。
4. 当 IP 地址无法映射到受支持语言时，系统应降级使用默认语言 `en`。
5. 系统应在 Next.js 中间件（`middleware.ts`）中实现 IP 检测逻辑，确保在边缘运行时（Edge Runtime）执行。
6. 当部署在 Vercel 或 Cloudflare 上时，系统应优先读取平台提供的地理位置标头（`x-vercel-ip-country`、`cf-ipcountry`）。

---

### 需求 4：货币本地化
**目标：** 作为最终用户，我希望在应用的定价页面和支付流程中看到与我所在地区匹配的货币，以便减少因货币换算带来的困惑。

#### 验收标准
1. 当用户地区被检测为中国大陆时，系统应在定价展示时优先显示人民币（CNY）价格（如适用）。
2. 当用户地区为欧盟成员国时，系统应显示欧元（EUR）价格。
3. 当用户地区为英国时，系统应显示英镑（GBP）价格。
4. 系统应在所有其他地区默认显示美元（USD）价格。
5. 货币格式化应使用 `Intl.NumberFormat` 并根据当前 locale 正确格式化（例如 `¥1,234.00`、`$1,234.00`）。
6. 当支付提供商不支持某种货币时，系统应降级使用美元（USD）并提示用户。

---

### 需求 5：支付方式地区适配
**目标：** 作为最终用户，我希望在结算时看到适合我所在地区的支付方式，以便能够使用熟悉的本地支付工具完成付款。

#### 验收标准
1. 当用户地区为中国大陆时，系统应在支付选项中展示支付宝（Alipay）和微信支付（WeChat Pay）（如 Stripe 支持）。
2. 当用户地区为欧盟时，系统应展示 SEPA 转账和 iDEAL 等本地支付方式（如 Stripe 支持）。
3. 系统现有的 `PaymentAdapter` 接口应扩展支持传入地区（locale/country）参数，以便适配器可返回地区特定的支付方式。
4. 当所在地区不支持特定支付方式时，系统应默认展示全球通用的信用卡/借记卡支付选项。
5. 地区支付方式的配置应集中管理在 `src/config/payments.ts` 中，与适配器实现解耦。

---

### 需求 6：hreflang 标签实现
**目标：** 作为 SEO 负责人，我希望所有多语言页面都包含正确的 hreflang 标签，以便搜索引擎能够正确索引各语言版本并降低内容重复惩罚风险。

#### 验收标准
1. 所有使用 `[locale]` 路由的页面应在 `<head>` 中自动生成对应所有支持语言的 `<link rel="alternate" hreflang="..." href="...">` 标签。
2. 系统应为每个多语言页面包含 `x-default` hreflang 标签，指向英文（`en`）版本。
3. hreflang 标签中的 URL 应使用完整的绝对路径（包含 `NEXT_PUBLIC_APP_URL` 域名）。
4. 当新增 `zh` locale 时，系统应自动在 hreflang 标签生成逻辑中包含 `zh` 语言变体。
5. 系统应提供可复用的工具函数生成 hreflang metadata，供各页面的 `generateMetadata` 使用。

---

### 需求 7：CDN/Edge 就近分发
**目标：** 作为平台运维，我希望静态资源和翻译文件通过 CDN/Edge 节点就近分发给用户，以便降低不同地区用户的访问延迟。

#### 验收标准
1. 系统应配置 Next.js `Cache-Control` 标头，确保翻译 JSON 文件和静态资源在 CDN 层可被缓存。
2. 当使用 Vercel 部署时，系统应在 `vercel.json` 中配置适当的缓存规则，使国际化路由能被边缘节点缓存。
3. 系统应为服务端渲染的国际化页面设置合理的 `stale-while-revalidate` 策略。
4. IP 地理位置检测逻辑应在 Edge Runtime 中执行，避免增加服务器端延迟。
5. 当 CDN 缓存命中时，应将 `Vary: Accept-Language` 标头纳入缓存键，以避免不同语言用户获得相同缓存内容。

---

### 需求 8：内容合规（GDPR 等）
**目标：** 作为合规负责人，我希望应用在面向不同地区用户时满足相关数据保护法规要求，以便规避法律风险。

#### 验收标准
1. 当用户来自欧盟（EU/EEA）地区时，系统应展示符合 GDPR 要求的 Cookie 同意横幅，在用户明确同意前不加载第三方追踪脚本。
2. 系统的隐私政策（`/privacy`）页面应提供多语言版本，至少包含 `en` 和 `zh` 两个语言版本。
3. 当用户请求删除账户时，系统应确保级联删除所有关联的个人数据（现有 Drizzle schema 的级联删除机制应覆盖此需求）。
4. 系统应在 `robots.txt` 中声明爬虫访问规则，并在适当时候限制对用户数据相关 API 的爬取。
5. 当收集用户 IP 用于地区检测时，系统不应将原始 IP 地址持久化到数据库，仅使用 IP 衍生的地区代码（country code）。
6. 系统应在 Cookie 同意管理中提供偏好设置选项，用户可分类选择允许的 Cookie 类型（必要/分析/营销）。
